# 고양이 동일 개체 후보 매칭 전환 문서

## 목표

- 촬영한 고양이를 현재 동네의 기존 고양이와 연결할 수 있는 후보 확인 흐름을 완성한다.
- `cat.id`는 판별 입력이 아니라 사용자가 동일 개체라고 확인한 뒤 기록을 묶는 식별자로만 사용한다.
- 잘못된 자동 병합보다 사용자가 중복 가능성을 검토할 수 있는 보수적인 흐름을 우선한다.
- 정확한 좌표를 저장하거나 후보 검색에 사용하지 않고 동네 단위 정보만 사용한다.

## 비목표

- 1단계에서 사진만으로 동일 개체를 자동 확정하지 않는다.
- 생성형 AI나 외부 유료 AI API를 도입하지 않는다.
- 검증되지 않은 감지 박스 크기와 이미지 해상도를 개체 유사도 점수로 사용하지 않는다.
- 기존 중복 고양이를 자동 병합하거나 삭제하지 않는다.

## 현재 문제

- iOS `CatVision`은 고양이 탐지와 누끼를 수행하지만 개체 재식별 임베딩을 생성하지 않는다.
- 현재 `featureVector`는 감지 신뢰도, 경계 상자 크기와 비율, 이미지 해상도로 구성되어 개체 구분에 사용할 수 없다.
- `cat_match_candidates` 테이블과 후보 UI는 있지만 후보를 생성하는 함수나 트리거가 없다.
- 후보가 비어 있으면 같은 고양이를 기존 `cat.id`에 연결하지 못하고 새 고양이로 중복 등록할 가능성이 있다.

## 1단계 범위

1. 촬영 관찰을 `cat_observations`에 저장한다.
2. 현재 동네와 연결된 기존 고양이를 최근 기록 순으로 후보화한다.
3. 사용자가 이미 수집한 고양이는 후보 상단에 우선 노출한다.
4. 후보 점수는 확률이 아니라 정렬을 위한 관련도 값으로만 사용한다.
5. 새 사진과 후보 대표 사진을 함께 보여주고 사용자가 동일 개체 여부를 최종 확인한다.
6. 사용자가 확인한 경우에만 기존 `cat.id`로 `record_cat_encounter`를 호출한다.
7. 후보가 없거나 확신하지 못하면 미확인 목격 또는 새 고양이 등록 흐름을 유지한다.

## 데이터 흐름

```text
촬영 및 고양이 탐지
  -> 원본/누끼 이미지 저장
  -> cat_observations 생성
  -> generate_cat_match_candidates RPC 호출
  -> 같은 동네의 기존 고양이 후보 저장
  -> 후보 1~5개 표시
  -> 사용자 확인
     -> 동일: 기존 cat.id로 만남 기록
     -> 신규: 새 cat.id 생성
     -> 불확실: cat_sightings에 미확인 목격 저장
```

## Supabase 변경

- `cat_match_candidates.match_method`에 후보 생성 방식을 기록한다.
- `cat_match_candidates.model_version`은 향후 개체 재식별 모델 버전을 기록할 수 있도록 둔다.
- `generate_cat_match_candidates` RPC는 인증 사용자 본인의 관찰에 대해서만 후보를 재생성한다.
- 후보 검색은 클라이언트가 전달한 현재 동네의 허용된 지역명 목록과 관찰의 지역명을 사용한다.
- `cat_match_candidates` 삭제 정책은 자신의 관찰에 속한 후보에만 허용한다.
- 새 공개 테이블은 만들지 않으며 기존 테이블의 RLS를 유지한다.
- RPC는 `security invoker`로 실행하고 `authenticated` 역할에만 실행 권한을 부여한다.

## 클라이언트 변경

- 관찰 저장 직후 후보 생성 RPC를 호출하고 생성된 후보를 조회한다.
- 현재 동네의 원본명과 데이터베이스 지역명을 함께 전달한다.
- 후보가 없을 때는 기존 고양이를 찾지 못했다는 상태를 명확히 표시한다.
- 후보 선택 시 즉시 연결하지 않고 동일 고양이 확인 대화상자를 표시한다.
- 화면 문구는 AI 확정, 인식 완료 같은 표현 대신 동네 기록 후보와 사용자 확인을 사용한다.

## 보안 및 개인정보

- 정확한 위도와 경도는 관찰 및 후보 데이터에 저장하지 않는다.
- 사용자는 다른 사용자의 관찰 또는 후보 행을 읽거나 수정할 수 없다.
- 클라이언트에는 `service_role` 또는 secret key를 노출하지 않는다.
- 공개 고양이와 공개 동네 기록만 후보 데이터의 입력으로 사용한다.

## 검증 항목

- 다른 사용자의 `observation_id`로 RPC를 호출하면 실패해야 한다.
- 같은 동네에 고양이가 있으면 최대 5개의 중복 없는 후보가 생성되어야 한다.
- 같은 RPC를 다시 호출해도 후보가 중복되지 않고 최신 순위로 교체되어야 한다.
- 다른 동네 고양이는 후보에 포함되지 않아야 한다.
- 후보 선택 전에는 `user_cat_collections`와 `cat_encounters`가 변경되지 않아야 한다.
- 사용자 확인 후에만 기존 고양이의 만남 및 개인 수집 기록이 증가해야 한다.
- 후보 없음, 새 고양이 등록, 잘 모르겠어요 흐름이 모두 동작해야 한다.
- 클라이언트 TypeScript 타입 검사를 통과해야 한다.

## 향후 2단계

- 고양이 얼굴과 털무늬를 표현하는 개체 재식별 임베딩 모델을 iOS와 Android에 동일하게 적용한다.
- 고양이별 여러 기준 사진과 모델 버전이 포함된 임베딩 테이블을 추가한다.
- Supabase `pgvector` RPC로 현재 동네 후보를 벡터 유사도 순으로 정렬한다.
- 검증 데이터셋으로 임계값을 보정하기 전까지 모델은 후보 순위만 제공하고 자동 병합하지 않는다.
- 중복 신고와 관리자 검토를 거쳐 기존 외래 키를 안전하게 이전하는 병합 이력을 추가한다.
