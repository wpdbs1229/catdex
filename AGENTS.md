# AGENTS.md

## 프로젝트 개요

이 저장소는 Supabase 기반 냥도감 앱입니다. 기존 NestJS 백엔드는 제거되었습니다.

- `catdex-client`: Expo React Native + TypeScript 모바일 앱
- `supabase`: Supabase Postgres, Auth, Storage, RLS, RPC 마이그레이션

제품 콘셉트는 사용자가 길고양이를 촬영하고 생물 도감처럼 수집하는 앱입니다. B2C 앱은 촬영, 기록, 수집, 개인 지도, 기본 MY 경험에 집중합니다.

## 제품/데이터 방향

- 기준 문서는 `docs/catdex-product-plan.md`입니다.
- 사용자가 명시적으로 동의한 촬영/관찰 기록은 정확한 GPS/EXIF 위치를 내부 민감 데이터로 저장할 수 있습니다.
- 정확 위치는 일반 사용자 화면, 지도, 공유 화면, 공개 API에 직접 노출하지 않습니다.
- 일반 공개값은 동, 구, 격자, 반경 흐림 등 지역 단위로 제공합니다.
- 정확 위치 저장을 도입할 때는 별도 전환 문서, Supabase 마이그레이션, RLS, 제한 RPC, 역할 기반 권한, 접근 목적 기록, 감사 로그 설계를 먼저 확정합니다.
- 이미지 파일을 외부에 공개하거나 Storage에 저장할 때는 위치 EXIF가 그대로 공개되지 않도록 제거하거나 공개용 파생본을 사용합니다.
- B2B, B2G, TNR, 구조, 연구 기능은 같은 데이터 기반을 공유하되 일반 앱 안에 모두 넣지 않고 별도 운영자/기관/지자체 대시보드 또는 권한 모드로 분리합니다.

## 작업 규칙

- 클라이언트 코드와 Supabase 코드는 서로 다른 폴더에 둡니다.
- 하나의 큰 파일에 몰아넣지 말고 기능/도메인 기반 폴더 구조를 우선합니다.
- 신규 백엔드 로직은 NestJS에 추가하지 않고 Supabase 마이그레이션, RLS, RPC, Storage 정책으로 구현합니다.
- 클라이언트는 백엔드 REST API 대신 `@supabase/supabase-js` 기반 접근을 사용합니다.
- Supabase `service_role` 또는 secret key는 클라이언트에 노출하지 않습니다.
- `public` 스키마에 생성하는 테이블은 RLS를 활성화하고 실제 접근 모델에 맞는 정책을 둡니다.
- 위치 데이터 작업은 위 제품/데이터 방향을 따릅니다.

## 현재 앱 표면

- 하단 탭은 `홈`, `도감`, `촬영`, `MY` 4개입니다.
- `홈`은 개인 발견 지도 화면입니다.
- `도감`은 표지 없이 2열 고양이 카드 그리드로 시작합니다.
- 검색과 필터는 도감 상단 필터 아이콘을 눌렀을 때만 열립니다.
- `MY`는 프로필, 수집/재발견 통계, 최근 만난 고양이, 탐험 기록, 알림 설정, 로그아웃을 제공합니다.
- 장식, 결제, 공개 탐색, 반응, 획득 보상 기능은 현재 클라이언트와 DB 활성 모델에서 제거된 범위입니다.

## 진행 방식

- 큰 기능은 별도 전환 문서를 먼저 만들고, 문서의 단계 순서대로 구현합니다.
- 위치/관찰 데이터 구조를 바꾸는 기능은 `docs/catdex-product-plan.md`의 위치 정책과 B2C/B2B/B2G 표면 분리 원칙부터 확인합니다.
- 단계마다 구현 후 검증하고, 오류가 발견되면 수정 후 다시 검증합니다.
- 기능 작업은 기능마다 별도 브랜치를 만들어 진행합니다.
- 커밋은 가능한 가장 작은 논리 단위로 나눕니다.
- 기능 단위 작업이 끝나면 PR을 생성합니다.
- 커밋/PR을 만들 수 없는 환경이면 즉시 사용자에게 알리고, 코드 변경 범위를 더 키우기 전에 저장소 상태를 먼저 정리합니다.

## GitHub 운영 규칙

- `main` 브랜치는 운영 배포 기준 브랜치로 사용합니다.
- `dev` 브랜치는 개발 및 통합 검증 기준 브랜치로 사용합니다.
- 기능 추가, 버그 수정, 리팩터링 등 실제 작업은 항상 별도 작업 브랜치에서 진행합니다.
- `main` 또는 `dev` 브랜치에서 직접 코드를 수정하거나 커밋하지 않습니다.
- 개발 검증이 끝난 작업 브랜치는 `dev` 브랜치로 PR을 생성합니다.
- 운영 반영은 `dev`에서 충분히 검증된 뒤 `main`으로 PR을 생성합니다.

## PR 및 커밋 작성 규칙

- PR 제목, PR 본문, 커밋 메시지는 항상 한글로 작성합니다.
- 변경 내용은 사용자가 바로 이해할 수 있도록 간단하고 구체적으로 작성합니다.
- PR 본문에는 변경 요약, 검증한 내용, 참고할 점 또는 남은 이슈를 포함합니다.

## 개발/검증 워크플로

- 클라이언트 작업은 기본적으로 `catdex-client` 디렉터리에서 진행합니다.
- 패키지 매니저는 `npm` 기준입니다.
- Node.js 버전은 `>=20.19.4 <21 || >=22` 제약을 따릅니다.
- 기본 검증은 `cd catdex-client && npm run typecheck`입니다.
- 로컬 실행이 필요하면 `cd catdex-client && npm run start`를 기본으로 사용합니다.
- 디바이스 연결 방식에 따라 `npm run start:lan`, `npm run start:tunnel`을 사용합니다.
- 에뮬레이터 실행은 `npm run android`, `npm run ios`를 사용합니다.
- OAuth/딥링크 검증 시 `catdex-client/.env.example`에 맞춰 `EXPO_PUBLIC_OAUTH_REDIRECT_URI`를 설정합니다.
- Supabase 환경변수는 `catdex-client/.env.example` 기준으로 `EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY`를 설정합니다.
- Supabase 로컬 CLI 실행/마이그레이션 적용 명령은 아직 프로젝트 기준으로 확정되지 않았습니다.
