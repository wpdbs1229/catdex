# AGENTS.md

## 프로젝트 개요

이 저장소는 Supabase 기반 앱입니다. 기존 NestJS 백엔드는 제거되었습니다.

- `catdex-client`: Expo React Native + TypeScript 모바일 mockup 앱입니다.
- `supabase`: Supabase Postgres, Auth, Storage, RLS, RPC 마이그레이션을 관리합니다.

제품 콘셉트는 "냥도감"입니다. 사용자가 길고양이를 촬영하고, 생물 도감처럼 수집하는 앱입니다. B2C 앱은 쉽고 귀여운 기록/수집 경험을 중심으로 유지하되, 장기적으로 관찰 데이터가 TNR, 돌봄, 지자체, 연구 협업에 활용될 수 있는 구조를 지향합니다.

## 제품/데이터 방향

- 기준 문서는 `docs/catdex-product-plan.md`입니다. 초기 비전 문서의 방향을 반영해 "정확 위치는 내부 자산, 공개 위치는 흐림 정보"를 원칙으로 합니다.
- 사용자가 명시적으로 동의한 촬영/관찰 기록은 정확한 GPS/EXIF 위치를 내부 민감 데이터로 저장할 수 있습니다.
- 정확 위치는 일반 사용자 화면, 공개 도감, 지도, 공유 화면, 공개 API에 직접 노출하지 않습니다. 일반 공개값은 동/구/격자/반경 흐림 등 지역 단위로 제공합니다.
- 정확 위치 저장은 별도 전환 문서와 Supabase 마이그레이션, RLS, RPC, 감사 로그 설계를 먼저 확정한 뒤 진행합니다.
- 정확 위치는 private schema, 제한 RPC, 역할 기반 권한, 접근 목적 기록, 접근 로그를 전제로 설계합니다.
- 이미지 파일을 외부에 공개하거나 Storage에 저장할 때는 위치 EXIF가 그대로 공개되지 않도록 제거하거나 공개용 파생본을 사용합니다.
- B2C 모바일 앱은 촬영, 기록, 수집, 꾸미기, 공유, 가벼운 관찰 태그 중심의 단순한 경험을 유지합니다.
- B2B, B2G, TNR, 구조, 연구 기능은 같은 데이터 기반을 공유하되, 일반 앱 안에 모두 넣지 않고 별도 운영자/기관/지자체 대시보드 또는 권한 모드로 분리합니다.
- TNR, 돌봄, 민원, 연구 목적 화면은 정확 위치 접근이 필요한 경우에만 역할, 목적, 감사 로그를 거쳐 제공하고, 기본 공개값은 흐림 위치입니다.
- 기존 문서나 코드 주석이 "정확한 좌표는 저장하지 않는다"는 방향을 전제한다면 레거시 전제로 보고, 새 작업에서는 이 섹션을 우선하며 관련 문서를 함께 갱신합니다.

## 작업 규칙

- 클라이언트 코드와 API 코드는 서로 다른 폴더에 둡니다.
- 하나의 큰 파일에 몰아넣지 말고, 기능/도메인 기반 폴더 구조를 우선합니다.
- 신규 백엔드 로직은 NestJS에 추가하지 않고 Supabase 마이그레이션, RLS, RPC, Storage 정책으로 구현합니다.
- 클라이언트는 백엔드 REST API 대신 `@supabase/supabase-js` 기반 접근을 사용합니다.
- Supabase `service_role` 또는 secret key는 클라이언트에 노출하지 않습니다.
- `public` 스키마에 생성하는 테이블은 RLS를 활성화하고 실제 접근 모델에 맞는 정책을 둡니다.
- 위치 데이터 작업은 위 `제품/데이터 방향`을 따릅니다. 정확 좌표는 동의 기반 내부 민감 데이터로만 저장하고, 공개/일반 API는 지역 단위 정보만 제공합니다.

## 진행 방식

- 큰 기능은 별도 전환 문서를 먼저 만들고, 문서의 단계 순서대로 구현합니다.
- 위치/관찰 데이터 구조를 바꾸는 기능은 `docs/catdex-product-plan.md`의 위치 정책과 B2C/B2B/B2G 표면 분리 원칙부터 확인합니다.
- 단계마다 구현 후 검증하고, 오류가 발견되면 수정 후 다시 검증합니다.
- 기능 작업은 기능마다 별도 브랜치를 만들어 진행합니다.
- 커밋은 가능한 가장 작은 논리 단위로 나눕니다.
- 기능 단위 작업이 끝나면 PR을 생성합니다.
- 커밋/PR을 만들 수 없는 환경이면 즉시 사용자에게 알리고, 코드 변경 범위를 더 키우기 전에 저장소 상태를 먼저 정리합니다.
- 공유 도감 전환은 `docs/shared-catdex-migration.md`를 기준 문서로 삼습니다.
- 기능을 검증하고 오류가 생기면 수정합니다. 오류가 없다면, 브랜치를 분리하고 한글로 커밋 푸쉬 PR 절차를 진행합니다.

## GitHub 운영 규칙

- `main` 브랜치는 운영 배포(prd) 기준 브랜치로 사용합니다.
- `dev` 브랜치는 개발 및 통합 검증 기준 브랜치로 사용합니다.
- 기능 추가, 버그 수정, 리팩터링 등 실제 작업은 항상 별도 작업 브랜치에서 진행합니다.
- 각 기능별 브랜치는 하나의 작업 단위로 봅니다. Codex 스레드 하나가 작업공간 하나처럼 동작하도록, 한 스레드에서는 가능한 한 하나의 작업 브랜치만 다룹니다.
- 브랜치 이름은 작업 성격이 드러나게 작성합니다. 예: `feature/login`, `fix/oauth-error`, `refactor/member-service`.
- `main` 또는 `dev` 브랜치에서 직접 코드를 수정하거나 커밋하지 않습니다. 필요한 경우 먼저 작업 브랜치를 만듭니다.
- 개발 검증이 끝난 작업 브랜치는 `dev` 브랜치로 PR을 생성합니다.
- 운영 반영은 `dev`에서 충분히 검증된 뒤 `main`으로 PR을 생성해 진행합니다.

## PR 및 커밋 작성 규칙

- PR 제목, PR 본문, 커밋 메시지는 항상 한글로 작성합니다.
- 변경 내용은 사용자가 바로 이해할 수 있도록 간단하고 구체적으로 작성합니다.
- PR 본문에는 최소한 다음 내용을 포함합니다.
  - 변경 요약
  - 검증한 내용
  - 참고할 점 또는 남은 이슈
- 불필요하게 긴 설명은 피하고, 무엇이 바뀌었는지와 왜 바뀌었는지가 드러나게 작성합니다.

## 개발/검증 워크플로

- 클라이언트 작업은 기본적으로 `catdex-client` 디렉터리에서 진행합니다.
- 패키지 매니저는 현재 `npm` 기준입니다. 새 스크립트나 의존성 문서화도 `npm` 기준으로 맞춥니다.
- Node.js 버전은 `>=20.19.4 <21 || >=22` 제약을 따릅니다.
- 구현 중 기본 검증은 `cd catdex-client && npm run typecheck`로 수행합니다.
- 화면/수동 QA가 필요한 단계는 `docs/shared-catdex-migration.md` 기준으로 `홈`, `도감`, `촬영`, `지도`, `MY` 화면을 확인합니다.
- 로컬 실행이 필요하면 `cd catdex-client && npm run start`를 기본으로 사용합니다.
- 디바이스 연결 방식에 따라 `cd catdex-client && npm run start:lan`, `cd catdex-client && npm run start:tunnel`을 사용합니다.
- 에뮬레이터 실행은 `cd catdex-client && npm run android`, `cd catdex-client && npm run ios`를 사용합니다.
- OAuth/딥링크 검증 시 `catdex-client/.env.example`에 맞춰 `EXPO_PUBLIC_OAUTH_REDIRECT_URI`를 설정합니다. 현재 예시는 Expo Go Android 에뮬레이터 `exp://127.0.0.1:8081`, 개발 빌드 `catdex://`입니다.
- Supabase 환경변수는 `catdex-client/.env.example` 기준으로 `EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY`를 설정합니다.
- TODO: 저장소에서 합의된 Supabase 로컬 CLI 실행/마이그레이션 적용 명령은 아직 확인하지 못했습니다. 확인되면 이 문서에 추가합니다.

각 단계는 구현 후 검증하고, 오류가 있으면 수정 후 재검증합니다. 단계 상태는 작업하면서 갱신합니다.

## 냥꾸러미 구독 기능 구현 순서

구독 기능은 "컬렉션 꾸미기"를 중심으로 시작합니다. 무료 사용자의 핵심 수집 경험은 막지 않고, 결제 혜택은 도감 표현, 장식, 공유, 시즌성 보상에 집중합니다.

### 명명 기준

- 구독명: `냥꾸러미`
- 꾸미기 메뉴명: `고양이 서랍`
- 배지명: `골목 배지`
- 시즌 스탬프명: `냥발 도장`
- 대표 고양이 영역: `우리 도감 주인공`

### 0단계: 전환 문서 작성

- `docs/nyangkkureomi-subscription.md` 문서를 먼저 만든다.
- 문서에는 목표, 비목표, 무료/구독 경계, 화면 목록, Supabase 스키마, RLS 정책, 클라이언트 작업 범위, 검증 항목, 출시 후 확장 항목을 정리한다.
- 이 단계에서는 구현하지 않고 기능 범위와 순서만 확정한다.
- 검증: 문서가 `냥꾸러미`, `고양이 서랍`, `골목 배지`, `냥발 도장`, `우리 도감 주인공` 명칭을 일관되게 사용하는지 확인한다.

### 1단계: 구독 권한 모델 설계

- Supabase에 사용자별 구독 상태를 저장하는 테이블을 설계한다.
- 클라이언트에는 `service_role` 또는 결제 검증용 secret key를 노출하지 않는다.
- MVP에서는 실제 스토어 결제 전까지 `free`, `nyangkkureomi` 같은 내부 entitlement 상태로 기능 게이트를 구현할 수 있다.
- 실제 결제 검증은 추후 Edge Function 또는 안전한 서버 측 검증 경로로 분리한다.
- 검증: RLS가 활성화되어 있고 사용자는 자신의 구독 상태만 읽을 수 있어야 한다.

### 2단계: 꾸미기 데이터 모델 추가

- Supabase 마이그레이션으로 다음 도메인을 추가한다.
  - 도감 프로필: 사용자 도감 소개, 선택한 표지, 공개용 문구
  - 고양이 서랍: 사용자가 보유하거나 선택한 꾸미기 아이템
  - 골목 배지: 배지 정의와 사용자 획득 내역
  - 냥발 도장: 시즌 스탬프 정의와 사용자 획득 내역
  - 우리 도감 주인공: 대표 고양이 설정
- 무료/구독 제한은 데이터 접근 제한보다 "선택 가능 개수"와 "프리미엄 아이템 사용 가능 여부"로 처리한다.
- 검증: 모든 public 테이블에 RLS를 켜고, 사용자가 자기 데이터만 생성/수정/삭제할 수 있는지 확인한다.

### 3단계: Seed와 mock 데이터 정리

- 개발 확인용 기본 표지, 기본 골목 배지, 시즌 냥발 도장 seed 데이터를 만든다.
- 클라이언트 mock 데이터가 필요한 경우 `catdex-client/src/shared/data` 아래에 기능별 파일로 분리한다.
- 프리미엄 아이템에는 `requiresSubscription` 같은 명시적 플래그를 둔다.
- 검증: 무료 사용자에게 기본 아이템이 보이고, 프리미엄 아이템은 잠금 상태로 보이는지 확인한다.

### 4단계: 클라이언트 기능 폴더 구성

- `catdex-client/src/features/subscription` 또는 `catdex-client/src/features/collection-customization` 아래에 기능을 분리한다.
- 화면, 컴포넌트, hook, 타입, API 접근 코드를 한 파일에 몰아넣지 않는다.
- Supabase 접근은 기존 `src/shared/supabase/client.ts`와 shared API 패턴을 따른다.
- 검증: TypeScript 타입 검사를 통과해야 한다.

### 5단계: 고양이 서랍 화면 구현

- `고양이 서랍`에서 표지, 프로필 장식, 골목 배지, 냥발 도장을 탭 또는 섹션으로 보여준다.
- 프리미엄 아이템은 잠금 표시와 `냥꾸러미` 안내를 함께 보여준다.
- 무료 사용자는 기본 표지와 기본 배지를 선택할 수 있어야 한다.
- 구독 사용자는 프리미엄 표지, 배지 진열, 시즌 도장 장식을 선택할 수 있어야 한다.
- 검증: 작은 화면에서도 텍스트가 겹치지 않고, 선택 상태와 잠금 상태가 명확해야 한다.

### 6단계: 우리 도감 주인공 구현

- 사용자가 수집한 고양이 중 대표 고양이를 설정한다.
- 무료 사용자는 1마리, `냥꾸러미` 사용자는 최대 3마리까지 설정한다.
- 대표 고양이는 마이페이지와 도감 표지에 노출한다.
- 검증: 존재하지 않거나 다른 사용자의 고양이를 대표로 설정할 수 없어야 한다.

### 7단계: 골목 배지와 냥발 도장 획득 로직

- 초기 MVP는 클라이언트 표시 중심으로 시작하되, 실제 획득 판정은 Supabase RPC 또는 DB 함수로 이동할 수 있게 설계한다.
- 예시 골목 배지: 첫 만남, 단골 냥이, 치즈냥 수집가, 동네 탐험가, 겨울 기록자.
- 예시 냥발 도장: 봄 산책, 여름 그늘, 가을 골목, 겨울 담요.
- 검증: 동일 배지가 중복 지급되지 않고, 시즌 기간이 지난 도장은 임의 획득되지 않아야 한다.

### 8단계: 냥꾸러미 안내와 업셀 화면

- `냥꾸러미` 안내는 기능을 가로막는 강한 paywall보다 잠금 아이템을 눌렀을 때 자연스럽게 보여준다.
- 가격 정책은 초기 기준으로 월 `3,900원`, 연 `39,000원`을 우선 문구에 반영하되, 실제 스토어 상품 ID와 가격은 배포 전에 App Store/Play Console 기준으로 확정한다.
- 안내 문구는 "수집 제한 해제"가 아니라 "내 도감을 더 예쁘게 꾸미기"에 맞춘다.
- 검증: 무료 핵심 플로우인 촬영, 등록, 기본 도감 조회가 결제 화면 때문에 막히지 않아야 한다.

### 9단계: 결제 연동 준비

- Expo 환경에서 사용할 결제 라이브러리와 스토어 상품 ID 전략을 정한다.
- 실제 결제 영수증 검증은 클라이언트 단독으로 처리하지 않는다.
- RevenueCat 같은 결제 관리 도구를 붙일 경우에도 공개 키와 secret key의 경계를 문서화한다.
- 검증: 개발/스테이징/운영 상품 ID가 섞이지 않도록 환경 변수를 분리한다.

### 10단계: 최종 검증, 브랜치, 커밋, PR

- 구현 단계마다 Supabase 마이그레이션 검증과 클라이언트 타입 검사를 실행한다.
- Expo 화면은 주요 화면 스크린샷 또는 수동 확인 결과를 남긴다.
- 오류가 있으면 수정 후 같은 검증을 다시 수행한다.
- 오류가 없으면 기능 브랜치를 분리하고, 한글 커밋 메시지로 작은 논리 단위 커밋을 만든다.
- 푸시와 PR 생성이 가능한 환경이면 PR을 생성한다.
- 커밋/푸시/PR이 불가능하면 즉시 사용자에게 알리고 추가 변경을 키우기 전에 저장소 상태를 정리한다.
