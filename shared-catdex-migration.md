# Shared Catdex Migration Plan

## 목표

냥도감을 사용자별 개인 도감에서 동네 사용자들이 함께 만드는 공유 도감으로 전환한다.

핵심 원칙:

- 고양이 자체는 공유 엔티티로 관리한다.
- 사용자의 수집 여부와 발견 기록은 사용자별로 관리한다.
- 정확한 위치는 저장하지 않고 지역 단위만 저장한다.
- 모든 public 테이블은 RLS를 활성화한다.
- 구현은 단계별로 진행하고 각 단계마다 검증한다.
- 커밋은 가장 작은 논리 단위로 나누고, 기능 단위 완료 후 PR을 만든다.

## 현재 상태

- `cats`는 `user_id`를 가진 사용자별 개인 고양이 테이블이다.
- `cat_encounters`는 사용자별 발견 기록이다.
- `region_cats`도 사용자별 지역-고양이 연결이다.
- RLS가 `auth.uid() = user_id` 기준이라 다른 사용자의 고양이는 보이지 않는다.
- 결과적으로 현재 앱은 공유 도감이 아니라 개인 도감이다.

## 목표 기능

1. 공유 고양이 등록
   - 로그인 사용자는 새 고양이를 공유 도감에 등록할 수 있다.
   - 최초 등록자는 `created_by`로 기록한다.
   - 등록 시 첫 발견 기록도 함께 생성한다.

2. 기존 고양이 재발견
   - 공유 고양이 상세에서 "나도 봤어요"로 발견 기록을 추가한다.
   - 발견 기록은 `cat_encounters`에 사용자별로 저장한다.
   - 공유 고양이의 총 발견 수와 최근 발견일을 갱신한다.

3. 내 도감
   - 공유 도감 전체와 별개로 내가 발견한 고양이만 모아 보여준다.
   - `user_cat_collections`로 사용자별 수집 상태를 관리한다.

4. 지역 도감
   - 지역별로 발견된 공유 고양이 목록을 보여준다.
   - 지역 중심점과 반경만 사용하고 정확한 좌표는 저장하지 않는다.

5. 사진
   - 고양이 대표 사진과 발견 기록 사진을 분리한다.
   - 공개 사진은 공유 조회 가능하게 하되, 삭제/수정은 업로더 또는 관리자만 가능하게 한다.

6. 신고/관리
   - 중복 고양이, 부적절한 사진, 위치 노출 위험, 잘못된 정보 신고를 받을 수 있게 한다.
   - 관리자 병합/숨김/삭제는 후속 단계로 둔다.

## 데이터 모델 방향

공유 테이블:

- `cats`: 공유 고양이
- `cat_encounters`: 발견 기록
- `user_cat_collections`: 사용자별 수집 상태
- `regions`: 지역
- `cat_regions`: 공유 고양이-지역 연결
- `cat_photos`: 고양이/발견 사진
- `reports`: 신고

주요 변경:

- `cats.user_id`를 제거하고 `cats.created_by`로 변경한다.
- `cats`는 로그인 사용자가 모두 조회할 수 있게 한다.
- `cat_encounters.user_id`는 유지한다.
- 내 도감은 `user_cat_collections`에서 계산한다.
- 지역별 목록은 `cat_regions` 기준으로 계산한다.

### 1. 공유 도감 스키마 마이그레이션 작성

상태: `[pending]`

- 기존 개인 도감 구조를 공유 도감 구조로 바꾸는 Supabase migration을 추가한다.
- `cats.created_by`, `user_cat_collections`, `cat_regions`, `cat_photos`, `reports`를 설계한다.
- 기존 데이터가 있다면 가능한 범위에서 마이그레이션한다.

검증:

- SQL 문법 검토.
- RLS 정책이 공유 조회와 개인 쓰기 경계를 정확히 나누는지 확인.
- `npm run typecheck` 전에 클라이언트 타입 변경 범위를 확인.

### 2. RPC 함수 전환

상태: `[pending]`

- `create_cat`은 공유 고양이 생성 + 첫 발견 기록 + 내 수집 추가를 한 트랜잭션으로 처리한다.
- `record_cat_encounter`는 공유 고양이에 발견 기록을 추가하고 내 수집 상태를 갱신한다.
- 중복 후보 조회 RPC는 후속 단계로 분리한다.

검증:

- 인증 없는 호출은 실패해야 한다.
- 생성 후 `cats`, `cat_encounters`, `user_cat_collections`, `cat_regions`가 함께 반영되어야 한다.

### 3. 클라이언트 데이터 조회 전환

상태: `[pending]`

- 홈/도감은 공유 고양이 목록을 조회한다.
- MY/내 도감은 `user_cat_collections` 기준으로 조회한다.
- 지역 지도는 모든 사용자의 공유 지역 고양이를 보여준다.

검증:

- `npm run typecheck` 통과.
- 로그인 사용자 기준 홈, 도감, 지도, MY 화면이 빈 데이터에서도 깨지지 않아야 한다.

### 4. 수집 UX 전환

상태: `[pending]`

- 등록 화면에서 새 공유 고양이 등록과 기존 고양이 재발견 흐름을 구분한다.
- 고양이 상세에 "나도 봤어요" 액션을 연결한다.

검증:

- 새 고양이 등록 후 공유 도감과 내 도감에 모두 반영되어야 한다.
- 재발견 후 발견 횟수와 최근 발견일이 갱신되어야 한다.

### 5. 공개 범위와 신고 기능

상태: `[pending]`

- 발견 기록 공개 필드를 제한한다.
- 신고 테이블과 신고 생성 API를 추가한다.
- 관리자 처리는 후속 PR로 분리한다.

검증:

- 본인이 아닌 사용자의 민감 정보가 노출되지 않아야 한다.
- 신고는 로그인 사용자만 생성할 수 있어야 한다.

### 6. 최종 시뮬레이터 검증

상태: `[pending]`

- 사용자 A/B 시나리오를 수동 또는 SQL seed로 검증한다.
- Expo 시뮬레이터에서 홈, 도감, 촬영, 지도, MY를 확인한다.

검증:
